# V5.3 Opportunity Intelligence V3.2.1.1 Google Maps Source Preservation Fix Report

## Objective

Fix the remaining Customer List source mapping bug where Google Maps imported customers were displayed as:

- Source: Manual Import

Expected result:

- Source: Google Maps

This fix does not change Website Import, Manual Import, Instagram, Facebook, or Search Result behavior.

## Root Cause

The customer source mapping logic only preserved Google Maps when the `source` field explicitly matched Google Maps.

Some Google Maps imported customers can arrive with:

- `source` defaulted to `Manual`
- `customer_source` defaulted to `Manual Import`
- a populated `google_maps_url`
- or a Google Maps URL in `source_url`

In that case, the system did not use the Google Maps URL evidence and kept `customer_source` as `Manual Import`.

- `src/server.mjs`

## Fix Details

### 1. Runtime Source Mapping

Updated `customerSourceValue()` to detect Google Maps evidence from:

- `source`
- fallback source
- `google_maps_url`
- `source_url`

Recognized Google Maps patterns include:

- `Google Maps`
- `google_maps`
- `googlemaps`
- `google.com/maps`
- `maps.app.goo.gl`
- `goo.gl/maps`
- `g.page`

If Google Maps evidence exists, `customer_source` is set to:

- `Google Maps`

### 2. Existing Data Repair

Updated startup/migration safety logic to repair existing records where:

- `customer_source = Manual Import`
- but Google Maps URL evidence exists

Those records are updated to:

- `customer_source = Google Maps`

### 3. Migration Update

Updated `024_v53_customer_source.sql` so new deployments and migrated environments also preserve Google Maps source from URL evidence.

### 4. Regression Tests

Added a regression test for the exact failure pattern:

- create customer with `source = Manual`
- include a `google_maps_url`
- verify returned `customer_source = Google Maps`
- verify Customer List source filter can find the record under Google Maps
