## Business Position

Search Result represents a discovered company candidate produced by a future Search Execution Engine.

It sits between:

AI Search Task

↓

Search Results

↓

Human Review

↓

Customer List

↓

AI Customer / Opportunity Intelligence

Search Result is not the final customer record. It is a reviewable discovery candidate.

## Entity Design

Entity name:

`Search Result`

Recommended table name:

`search_results`

Primary relationships:

- Belongs to `search_tasks`
- May convert into `customers`
- May create `customer_contacts`
- May reference AI execution logs and cost logs later

## Recommended Status Values

Search Result status should be review-oriented:

- `Draft`
- `Discovered`
- `Needs Review`
- `Qualified`
- `Rejected`
- `Converted`
- `Duplicate`

Recommended usage:

- `Draft`: created but incomplete.
- `Discovered`: captured from search task and ready for scoring.
- `Needs Review`: missing required fields or low confidence.
- `Qualified`: approved as useful prospect but not yet converted.
- `Rejected`: not relevant.
- `Converted`: converted into Customer List.
- `Duplicate`: matched existing Customer or existing Search Result.

## Recommended PostgreSQL-Compatible Table Design

```sql
CREATE TABLE search_results (
  id BIGSERIAL PRIMARY KEY,
  search_task_id BIGINT NOT NULL REFERENCES search_tasks(id) ON DELETE CASCADE,
  customer_id BIGINT REFERENCES customers(id) ON DELETE SET NULL,

  source_name TEXT,
  source_url TEXT,
  source_record_id TEXT,
  source_confidence INTEGER NOT NULL DEFAULT 50,
  raw_snapshot JSONB NOT NULL DEFAULT '{}'::jsonb,

  company_name TEXT NOT NULL,
  website TEXT,
  country TEXT,
  state TEXT,
  city TEXT,
  business_type TEXT,
  customer_type TEXT,

  contact_person TEXT,
  job_title TEXT,
  email TEXT,
  phone TEXT,
  linkedin_url TEXT,
  instagram_url TEXT,
  facebook_url TEXT,

  company_size TEXT,
  years_operating INTEGER,
  number_of_locations INTEGER,
  hospitality_experience TEXT,
  import_experience TEXT,

  opportunity_score INTEGER NOT NULL DEFAULT 0,
  score_level TEXT NOT NULL DEFAULT 'Low',
  score_reasons JSONB NOT NULL DEFAULT '[]'::jsonb,
  score_confidence INTEGER NOT NULL DEFAULT 0,
  scoring_profile JSONB NOT NULL DEFAULT '{}'::jsonb,
  scored_at TIMESTAMPTZ,

  product_match JSONB NOT NULL DEFAULT '[]'::jsonb,
  recommended_approach TEXT,
  notes TEXT,
  status TEXT NOT NULL DEFAULT 'Draft',

  created_by BIGINT REFERENCES users(id) ON DELETE SET NULL,
  reviewed_by BIGINT REFERENCES users(id) ON DELETE SET NULL,
  converted_by BIGINT REFERENCES users(id) ON DELETE SET NULL,
  reviewed_at TIMESTAMPTZ,
  converted_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_search_results_task_status
  ON search_results(search_task_id, status, opportunity_score DESC);

CREATE INDEX idx_search_results_company_location
  ON search_results(company_name, country, city);

CREATE INDEX idx_search_results_customer_type
  ON search_results(customer_type, score_level, opportunity_score DESC);

CREATE INDEX idx_search_results_customer_id
  ON search_results(customer_id);
```

## Future API Design

These endpoints are recommended for the future Search Execution Engine phase. They are not implemented in V3.15.

### List Search Results

`GET /api/search-tasks/:taskId/results`

Response:

```json
{
  "task": {
    "id": 12,
    "task_name": "Search Task #012",
    "status": "Ready"
  },
  "summary": {
    "total": 100,
    "qualified": 42,
    "needs_review": 35,
    "duplicates": 8,
    "converted": 15
  },
  "results": []
}
```

### Create Search Result

`POST /api/search-tasks/:taskId/results`

Purpose:

Create one discovered company candidate manually or from a future provider.

### Update Search Result

`PUT /api/search-results/:id`

Purpose:

Human review and correction.

### Qualify Search Result

`POST /api/search-results/:id/qualify`

Purpose:

Mark result as qualified.

### Reject Search Result

`POST /api/search-results/:id/reject`

Purpose:

Reject irrelevant result.

### Convert Search Result to Customer

`POST /api/search-results/:id/convert-to-customer`

Purpose:

Create or link:

- `customers`
- `customer_contacts`

Rules:

- Do not overwrite existing customers silently.
- Run duplicate detection before conversion.
- Preserve source traceability.

## Duplicate Detection Design

Before converting a Search Result into Customer List, compare:

- Company Name
- Website domain
- Email domain
- Phone
- Country + City
- LinkedIn URL
- Instagram URL

Suggested result:

```json
{
  "duplicate_check": {
    "status": "possible_duplicate",
    "matches": [
      {
        "customer_id": 88,
        "company_name": "ABC Hospitality Design",
        "match_reason": "Same website domain",
        "confidence": 92
      }
    ]
  }
}
```

## Permission Design

Owner / Admin:

- View all Search Results
- Qualify / reject / convert
- See source traceability

Sales:

- View assigned or visible Search Results
- Qualify and convert if allowed by current sales permissions
- Cannot delete source history

VA:

- Can create or edit basic discovered company information
- Cannot override opportunity score
- Cannot bulk-convert without review

Designer:

- No default access unless future workflow requires it

## Integration With Existing Modules

### Search Tasks

`search_results.search_task_id` connects each discovered company to the AI-generated Search Task.

### Customer List

Search Results become Customer records only after human conversion.

### AI Customer Intelligence

After conversion, the existing Customer Intelligence process can score and prioritize the customer.

### Product Intelligence

Product matching should store category/product references, not duplicate product data.

### AI Cost Control

Future Search Execution should log:

- Search result extraction cost
- Scoring cost
- Duplicate detection cost

V3.15 does not add new cost calls.
