## 1. AI Architecture Foundation Document

### 1.1 Architecture Goal

AI Sales OS V5.3 Lite should sit above the existing Restaurant Setup Pro business system.

The AI layer must:

- reuse current Product, Customer, Quote, PI, Order, Pricing, and Cost Control foundations
- generate recommendations, drafts, analysis, and logs
- avoid becoming the source of truth
- require human review before business records are changed
- run through AI Cost Control before any paid provider call

Target pattern:

```text
Existing Business Modules
        ↓
AI Context Builder
        ↓
AI Service Layer / AI Business Brain
        ↓
AI Provider Adapter
        ↓
AI Execution Log + AI Cost Control
        ↓
Human Review / Apply
        ↓
Existing Business Modules
```

### 1.2 AI Service Layer Design

The AI Service Layer should become the shared execution layer for all future AI modules.

It should not be tied to one feature such as lead discovery, proposal generation, or product content.

Recommended responsibilities:

1. Receive a business action request.
2. Identify the module and action type.
3. Request the required context package from AI Context Builder.
4. Resolve the prompt template and prompt version.
5. Estimate cost through AI Cost Control.
6. Check cache and budget.
7. Route the request to the correct provider adapter.
8. Store execution logs.
9. Return a structured AI result.
10. Store result as a draft/recommendation/analysis, not as final business data.

Suggested future action names:

- `lead.discovery.analyze`
- `opportunity.score.v2`
- `solution.generate`
- `pricing.review`
- `quote.review`
- `pi.review`
- `message.generate`
- `product.context.summarize`
- `customer.context.summarize`

The AI Service Layer should expose one consistent internal contract:

```text
runAiAction({
  moduleName,
  actionName,
  entityType,
  entityId,
  contextType,
  promptTemplateKey,
  providerPreference,
  requireConfirmation,
  userId,
  options
})
```

Expected output:

```text
{
  status,
  provider,
  model,
  promptVersion,
  contextSnapshotId,
  executionLogId,
  estimatedCost,
  actualCost,
  cacheStatus,
  result,
  reviewRequired
}
```

### 1.3 AI Provider Adapter Design

The provider adapter should isolate all model-specific behavior.

Business modules should never call OpenAI, Gemini, Claude, Qwen, or any other provider directly.

Recommended provider adapter interface:

```text
AIProviderAdapter
  estimate(input)
  generateText(input)
  generateStructured(input)
  validateConfig()
  providerStatus()
```

Supported provider modes:

| Provider Mode | Purpose |
| --- | --- |
| `mock` | Default safe local development mode |
| `rules` | Deterministic rules engine fallback |
| `openai` | Future real LLM/image provider |
| `gemini` | Future optional LLM provider |
| `claude` | Future optional LLM provider |
| `qwen` | Future optional LLM provider |

Provider rules:

- Default provider must remain `mock` or `rules`.
- Paid providers require Admin/Owner activation.
- Missing API keys must not break the system.
- API keys must never be logged.
- Provider errors must be stored in AI logs and visible in Debug Center.
- Provider execution must be cancellable/retryable for long-running jobs in later phases.

### 1.4 AI Context Builder Design

The AI Context Builder prepares safe, structured context for AI execution.

It must pull from existing business modules without duplicating data.

Context types:

| Context Type | Source Modules | Example Use |
| --- | --- | --- |
| Product Context | Product Library, Product Intelligence, Product Knowledge, Price Engine | Recommend furniture package |
| Customer Context | Opportunity Intelligence, Contacts, Activity, Outreach | Score lead / draft message |
| Inquiry Context | Sales Intelligence, inquiry analysis, selected products | Generate solution draft |
| Quote Context | Quote Builder, quote items, custom items, price snapshots | Review quote / pricing risk |
| PI Context | PI Builder, company settings, payment/freight fields | Generate PI remarks / quality check |
| Business Rules Context | AI Cost Control, permissions, role rules, source-of-truth map | Govern AI action |

The Context Builder should produce:

1. machine-readable JSON
2. human-readable debug summary
3. source references
4. redacted context based on role/action
5. context hash for cache

Important redaction rules:

- Supplier cost may be included only for authorized pricing actions.
- Sales-facing recommendation context must not expose purchasing-only fields.
- Customer-facing PI/message context must not include internal notes unless explicitly approved.

Example context structure:

```text
{
  contextType: "quote",
  entityId: 123,
  sourceModules: ["Quote Builder", "Product Library", "Price Engine"],
  safeForCustomer: false,
  redactionLevel: "internal-sales",
  productFacts: [],
  customerFacts: {},
  quoteFacts: {},
  missingData: [],
  sourceReferences: []
}
```

### 1.5 Prompt Management Design

Prompt management should become a shared system layer.

Future prompt templates should include:

- prompt key
- module name
- action name
- prompt version
- provider compatibility
- input variables
- output schema
- cost profile
- owner/reviewer
- active/inactive status
- changelog

Prompt examples:

| Prompt Key | Purpose |
| --- | --- |
| `lead.discovery.website_analysis.v1` | Analyze a restaurant website for sales signals |
| `opportunity.score.explain.v1` | Explain opportunity grade and next action |
| `solution.generate.package.v1` | Generate furniture package draft |
| `pricing.review.margin_risk.v1` | Explain pricing risk and missing data |
| `pi.review.quality_check.v1` | Check PI readiness and wording |
| `sales.message.first_touch.v1` | Generate outreach draft |

Prompt rules:

- Prompt output should be structured whenever possible.
- Prompt versions must be preserved.
- Prompt changes must not silently alter existing approved outputs.
- Prompts must define whether output is internal-only or customer-facing.
- Customer-facing prompts require stricter review and safer context.

AI execution logs should make every AI result traceable.

Minimum log fields for future implementation:

- id
- module_name
- action_name
- entity_type
- entity_id
- user_id
- provider
- model
- prompt_template_key
- prompt_version
- context_snapshot_id
- input_hash
- output_snapshot
- input_tokens
- output_tokens
- estimated_cost_usd
- actual_cost_usd
- cache_status
- status
- error_message
- started_at
- completed_at

Current reusable foundations:

- `ai_cost_logs`
- `ai_cache_records`
- `customer_ai_analysis_runs`
- `ai_product_content_drafts`
- `ai_image_generation_tasks`

Future recommendation:

- If V5.3 AI actions become cross-module and more complex, add a unified `ai_execution_logs` table.
- Do not replace existing module-specific logs immediately.
- Link module-specific drafts/reviews to the unified execution log when added.

### 1.7 AI Cost Control Integration Point

Every future AI execution must pass through AI Cost Control.

Required sequence:

```text
User action
  ↓
AI Context Builder creates context
  ↓
AI Service Layer requests estimate
  ↓
AI Cost Control checks budget/cache/provider policy
  ↓
If needed: user confirmation
  ↓
Provider Adapter executes
  ↓
AI Cost Control records actual cost
  ↓
AI Execution Log stores result
  ↓
Result enters human review workflow
```

Cost control must apply to:

- AI Lead Discovery
- website analysis
- Opportunity Intelligence upgrade
- AI Solution Generator
- AI Pricing Intelligence
- AI PI Intelligence
- AI Sales Assistant
- Product content generation
- image generation
- bulk/batch operations

## 3. Data Flow Design

### 3.1 Core Rule

AI cannot become the source of truth.

AI can only create:

- recommendations
- drafts
- analysis
- logs
- snapshots
- review records

Official business records change only when an authorized user applies or approves an AI output.

### 3.2 Product Data Flow into AI

```text
Product Library / Product Intelligence / Product Knowledge
        ↓
AI Context Builder: Product Context
        ↓
AI Business Brain
        ↓
AI Solution Generator / AI PI Intelligence / AI Sales Assistant
        ↓
Recommendation or Draft
        ↓
Human Review
        ↓
Optional Apply to Quote / Product Intelligence
```

Allowed AI outputs:

- recommended products
- recommended categories
- solution draft items
- missing product data warnings
- product summary drafts
- PI description suggestions

Not allowed:

- direct Product Library overwrite
- direct price calculation override
- exposing supplier cost to unauthorized users

### 3.3 Customer Data Flow into AI

```text
Opportunity Intelligence / Contacts / Activity / Inquiries
        ↓
AI Context Builder: Customer Context
        ↓
AI Business Brain
        ↓
Opportunity Intelligence Upgrade / Sales Assistant
        ↓
Analysis, next action, outreach draft
        ↓
Human Review
        ↓
Optional update to customer status / activity / outreach draft
```

Allowed AI outputs:

- opportunity analysis
- data gap detection
- next action recommendation
- sales angle
- outreach draft
- lead priority explanation

Not allowed:

- automatic external outreach
- unreviewed contact overwrite from external sources
- untraceable lead enrichment

### 3.4 Quote Data Flow into AI

```text
Quote Builder
        ↓
AI Context Builder: Quote Context
        ↓
AI Pricing Intelligence / AI Solution Generator
        ↓
Risk review, missing data, recommended package changes
        ↓
Sales Review
        ↓
Optional manual quote update
```

Allowed AI outputs:

- quote completeness review
- price risk explanation
- missing product data warnings
- package alternatives
- internal sales notes

Not allowed:

- silent unit price changes
- silent quantity changes
- final calculation changes

### 3.5 PI Data Flow into AI

```text
PI Builder / Quote Builder / Company Settings
        ↓
AI Context Builder: PI Context
        ↓
AI PI Intelligence
        ↓
PI review, remarks, message drafts
        ↓
Sales Review
        ↓
Optional Apply to PI text fields / copy message
```

Allowed AI outputs:

- PI quality checklist
- payment wording
- freight wording
- general remarks
- email body
- WhatsApp message

Not allowed:

- changing grand total
- changing bank details
- changing official payment percentages without user input
- sending customer messages automatically

### 3.6 Lead Discovery Data Flow into AI

Future flow:

```text
External Source Connector
        ↓
Lead Discovery Batch
        ↓
AI Context / Signal Detection
        ↓
Draft Lead Record
        ↓
Human Review
        ↓
Approved Customer / Opportunity Intelligence
```

Allowed AI outputs:

- draft customer
- source confidence
- detected business type
- expansion/renovation/furniture signals
- recommended next action

Not allowed:

- creating sales-ready customer without review
- automatic outreach
- uncontrolled scraping

## 5. Phase 0 Architecture Decisions

### Decision 1 — AI Is Not Source of Truth

AI outputs must remain recommendations, drafts, analysis, logs, or snapshots.

### Decision 2 — AI Cost Control Is Mandatory

Every future provider execution must go through AI Cost Control before execution.

### Decision 3 — Existing Modules Stay Stable

Product Library, Quote Builder, PI Builder, Opportunity Intelligence, and Price Engine must not be rebuilt for V5.3.

### Decision 4 — Human Review Is Required

Customer-facing or business-record-changing AI output requires human review and explicit Apply/Approve.

### Decision 5 — Context Builder Comes Before Real AI Expansion

AI modules should not call business tables ad hoc. A shared context builder should define what data AI sees and why.

### Decision 6 — Provider Adapter Comes Before Paid AI Scale

OpenAI/Gemini/Claude/Qwen should be called only through a provider adapter, never directly from business modules.
