## 1. Current UI Problems

### 1.1 Import Center Looks Too Technical

The current Product Import Center exposes too much operational/debug-like information directly in the main user flow.

Problems:

- It feels like a parser/debug page instead of a business import workspace.
- Technical import details compete with product review decisions.
- Owner/Product Manager users need a simple answer: what was detected, what needs review, and what can be approved.
- Debug details should still exist, but they should be secondary or expandable.

### 1.2 Product Review Uses Long Vertical Cards

Current draft review is difficult for real supplier spreadsheets because:

- Long cards require too much scrolling.
- Variant comparisons are hard.
- Missing fields are not easy to scan in bulk.
- Approval decisions are slow.
- Product grouping and variant count are not visually obvious.

For real supplier import review, users need table/grid density, not long narrative cards.

### 1.3 Product Data Is Not Presented Like a Professional Product Backend

The current Product Library has working product management, but it does not yet fully behave like a SaaS-grade product intelligence workspace.

Missing UX qualities:

- Clear product health/readiness view.
- Image-first product recognition.
- Fast product list scanning.
- Professional variant grid.
- Clear separation between public sales data and internal supplier data.
- Better presentation of AI readiness, data quality, and source traceability.

### 1.4 Future AI/Quote/PI Needs Are Not Obvious in UI

The future Product Intelligence Center must support:

- AI recommendation
- Customer-product matching
- Quote generation
- PI generation
- Website product display
- AI Sales Assistant context

The UI should make these readiness gaps visible:

- Missing image
- Missing SKU
- Missing category
- Missing attributes
- Missing variant price
- Missing selling points
- Missing AI knowledge
- Missing quote-ready data
- Missing PI-ready customer-facing content

### 1.5 Permission Boundaries Need UI Clarity

Server-side redaction already protects supplier cost and internal data. PIC-2 UI must make permission boundaries clear:

- Owner/Admin sees internal supplier and cost data.
- Sales sees selling/reference information but not supplier confidential fields.
- VA can assist with drafts but cannot approve or view confidential purchasing data.
- Product Manager/Sales Admin can review product quality and import data based on configured role rules.

The UI should not show empty confusing gaps where redacted data was removed. It should show role-appropriate sections.

## 2. Target UI Architecture

### 2.1 Product Intelligence Center Positioning

The Product Intelligence Center should become a professional SaaS product management workspace for Restaurant Setup Pro AI Sales OS.

Reference direction:

- Shopify-style product management clarity.
- Alibaba supplier product data density.
- ERP product center structure.

But the UI must be designed specifically for Restaurant Setup Pro:

- Hospitality furniture products.
- Product/variant/catalog import workflows.
- AI-ready product context.
- Quote and PI readiness.
- Internal supplier confidentiality.

### 2.2 UI Design Principles

1. Product Library remains source of truth.
2. Product Intelligence UI is an operating interface, not a duplicate product system.
3. Desktop-first for business operations.
4. Image-first recognition for product users.
5. Table/grid-first review for imports.
6. Tabs for deep product detail.
7. AI recommends; human approves.
8. Supplier cost visibility follows permissions.
9. Quote/PI/AI readiness should be visible without opening every product.
10. Debug details should be available but not dominate the main workflow.

### 2.3 Proposed Navigation

PIC-2 should improve the existing Product Library / Product Intelligence area without adding unnecessary disconnected menus.

Recommended navigation model:

```text
Product Intelligence
  - Dashboard
  - Products
  - Import Review
  - Categories
  - Attributes
  - Variants
  - Price Rules
```

If the current app keeps Product Library as the menu group, then PIC-2 may label the landing page as “Product Intelligence Center” while reusing existing navigation routes.

Do not create a second product area that confuses users.

## 5. AI Assisted Review

### 5.1 AI Role

AI should assist product users, not replace product review.

AI may detect:

- Missing fields
- Duplicate products
- Wrong category
- Incorrect attribute mapping
- Unclear product/variant grouping
- Missing images
- Suspicious price/currency values
- Low confidence source rows

AI may recommend:

- Suggested category
- Suggested attributes
- Suggested variants
- Suggested duplicate action
- Suggested selling points
- Suggested AI keywords

AI must not:

- Automatically approve products.
- Automatically overwrite Product Library.
- Automatically change price.
- Expose supplier cost to unauthorized users.
- Create customer-facing content without review.

### 5.2 AI Recommendation UI Pattern

Every AI suggestion should show:

- Suggested value
- Current value
- Confidence
- Reason
- Source reference
- Accept / Edit / Reject actions

Suggested UI states:

- Suggested
- Accepted
- Edited
- Rejected
- Applied

### 5.3 Cost Control UI

If future AI analysis uses paid providers:

- Show estimated cost before batch AI analysis.
- Require confirmation when cost exceeds threshold.
- Show provider mode: Mock / Rules / Paid Provider.
- Show cache hit where applicable.

For PIC-2 UI implementation, this can be designed as future-ready. No provider call is required by this specification.

## 6. User Workflow

### 6.1 End-to-End Product Intelligence Workflow

```text
Supplier File
  ↓
AI / Smart Analysis
  ↓
Import Review
  ↓
Product Draft
  ↓
Human Approval
  ↓
Product Library
  ↓
AI Ready Product
```

### 6.2 Detailed Workflow

#### Step 1 — Upload Supplier File

User uploads:

- XLSX
- XLS
- CSV
- Future product catalog source

System shows:

- File name
- File type
- Sheet count
- Detected rows
- Detected images

#### Step 2 — Smart Analysis

System detects:

- Headers
- Columns
- Product rows
- Variant rows
- Images
- Supplier fields
- Currency/price fields

AI/rules may suggest mappings.

#### Step 3 — Import Review

User sees a table of product drafts.

User can:

- Filter Needs Review
- Edit fields
- Review variants
- Approve clean rows
- Reject wrong rows
- Split/merge product groups

#### Step 4 — Product Draft Approval

Approved drafts create or update existing Product Library records only after human confirmation.

#### Step 5 — Product Library

Approved products appear in Product Intelligence Product List.

#### Step 6 — AI Ready Product

Product becomes ready for downstream flows when:

- Product master data is complete.
- Attributes are sufficient.
- Variants/pricing are complete.
- Images are available.
- AI Knowledge is reviewed.
- Quote/PI required fields are ready.

## 8. Permission Considerations

### 8.1 Role-Based Visibility

PIC-2 UI must follow server-side permissions and should not attempt to reconstruct hidden data.

| Role | Product List | Supplier | Cost | Import Review | Approve | AI Knowledge |
| --- | --- | --- | --- | --- | --- | --- |
| Owner | Full | Visible | Visible | Full | Yes | Full |
| Admin | Full | Visible | Visible | Full | Yes | Full |
| General Manager | Full | Visible if enabled | Visible if enabled | Full | Yes | Full |
| Purchasing | Internal product view | Visible | Visible | Full | Yes if enabled | Internal |
| Sales Admin | Full product view | Limited | Limited by policy | Full | Yes if enabled | Review |
| Sales | Approved/sales-safe products | Hidden | Hidden supplier cost | View approved only | No | Approved AI knowledge |
| VA | Draft support | Hidden | Hidden | Upload/review drafts if allowed | No | Limited |
| Designer | Product/media support | Hidden | Hidden | Media-related if allowed | No | Draft/media support |

### 8.2 UI Rules for Redacted Fields

If API does not return a field:

- Do not show empty “Supplier Cost” placeholders to unauthorized users.
- Do not display “hidden” values that imply confidential data exists.
- Show role-appropriate summaries instead:
  - Sales: “Selling price available” or “Price requires confirmation.”
  - VA: “Internal purchasing data not available.”

### 8.3 Customer-Facing Safety

Customer-facing content must never include:

- Supplier cost
- Supplier notes
- Purchase cost history
- Profit margin
- Internal source file notes
- AI unreviewed claims

## 14. Risks

### 14.1 Duplicate UI Risk

Risk:

Creating another “Product Intelligence” UI beside existing Product Library may confuse users.

Mitigation:

PIC-2 should upgrade the existing product area rather than create a disconnected product system.

### 14.2 Supplier Cost Leakage Risk

Risk:

Pricing and source tabs could expose supplier cost to Sales/VA.

Mitigation:

Use API capability flags and server-side redaction. Do not render internal cost columns unless the API confirms permission.

### 14.3 Table Complexity Risk

Risk:

Product tables and import review grids can become visually overwhelming.

Mitigation:

Use:

- Default essential columns.
- Expandable details.
- Clear filters.
- Bulk actions.
- Saved views later.

### 14.4 AI Trust Risk

Risk:

Users may treat AI suggestions as final.

Mitigation:

Always label AI suggestions as suggestions. Require human approval for product data changes.

### 14.5 Performance Risk

Risk:

Large catalogs and many variants may slow frontend rendering.

Mitigation:

Use pagination, lazy detail loading, thumbnails, and avoid loading full product details on list page.

### 14.6 Existing Workflow Regression Risk

Risk:

Redesigning UI could break working Product Library, Import, Quote, or PI flows.

Mitigation:

Implement in phases. Keep existing APIs stable. Run full regression tests after each phase.

### 14.7 Monolithic Frontend Risk

Risk:

Adding more UI into a single frontend file increases maintenance cost.

Mitigation:

During implementation, separate render functions clearly. Plan a later frontend modularization phase if needed.
